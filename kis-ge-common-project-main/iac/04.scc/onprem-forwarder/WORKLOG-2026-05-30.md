## 3. 해결한 문제 체인 (시간순)

배포는 단순하지 않았고, 다음 연쇄를 순서대로 해소했다.

### 3-A. forwarder 본체
1. **IP 부족** — Direct VPC egress subnet이 `/28`이라 Cloud Run 배포 health check 실패
   (`error code 9 / no sufficient IP`). → **subnet `/28`→`/26`→`/24`** 확장 (in-place).
   - 교훈: Direct VPC egress는 인스턴스 + revision 롤아웃 IP가 필요해 `/26` 이상 필수.
2. **tainted / deletion_protection 데드락** — 배포 실패로 tainted → 강제 replace →
   replace의 destroy가 `deletion_protection=true`에 막힘. (untaint/unprotect/heal job으로 시도)
3. **진짜 원인 = `lifecycle.ignore_changes=[image]`** — placeholder(hello-app)에서
   실제 이미지로 갱신을 막아 서비스가 Ready=False에 갇힘. **제거**하니 image 변경이
   **in-place revision 업데이트(destroy 없음)** → deletion_protection/taint/replace 전부 무관.
   `heal-scc-forwarder` job은 불필요해져 제거.
4. **`-refresh=false` + push_endpoint** — apply가 stale `service.uri=""`로 plan해
   subscription push_endpoint가 `required`/`invalid`/`inconsistent final plan` 유발.
   → `apply-scc-onprem-forwarder`에서 `-refresh=false` 제거.
5. **Cloud Build 403** — 기본 빌드 SA(compute/cloudbuild)가 org 정책
   (`disable_automatic_iam_grants_default_sa`)으로 권한 0 → 스테이징 소스 read 실패.
   → `--service-account`로 **IaC SA**가 빌드 실행하도록 변경.

### 3-B. SCC notifications (발신원)
1. **topic 부재 (404)** — forwarder subscription이 없는 topic에 attach.
   `enable_phase6=false`라 topic 미생성. → **`enable_phase6=true`**.
2. **service agent lazy 생성 / FAILED_PRECONDITION** — `gcp-sa-scc-notification` 에이전트는
   첫 NotificationConfig 생성 시 만들어지고, v2 config 생성은 publisher 권한이 precondition.
   → 순서를 `topic → publisher IAM → (60s) → config`로 정렬.
3. **filter 문법** — `severity IN ["HIGH","CRITICAL"]` 미지원 → `(severity="HIGH" OR severity="CRITICAL")`.
4. **v1 API 종료** — `google_scc_notification_config`(v1) → **`google_scc_v2_organization_notification_config`** (location=global).
5. **409 already exists** — 이전 시도 중 생성된 config를 `imports.tf` import block으로 흡수.

## 8. 핵심 교훈

- **Cloud Run Direct VPC egress subnet은 `/26` 이상** (롤아웃 IP 여유). `/28`은 배포 실패.
- **초기 이미지 배포 시 `ignore_changes=[image]`는 함정** — placeholder에 갇힘. 태그 기반이면 terraform이 관리하게 두는 편이 단순.
- **이미지 업데이트는 in-place(새 revision)** — destroy가 아니라 `deletion_protection`/replace와 무관.
- **SCC notification은 v2 API** (`google_scc_v2_organization_notification_config`), filter는 `OR`(IN 미지원), publisher 권한이 config 생성 precondition, service agent는 lazy 생성.
- **org 자동 IAM 부여 차단 정책** 환경에선 Cloud Build를 **명시적 SA(`--service-account`)** 로 실행해야 함.
- **CI 수동 job은 반드시 최신 파이프라인(최신 커밋)에서 실행** — 옛 파이프라인은 옛 코드.
